# 一次开发，多轮发布

这份发布序列用于把已经完成的系统能力分批讲清楚。每一轮必须配一个真实演示、一个可验证文件或命令、一个明确的后端承接，不把旧功能伪装成新开发。

## 发布原则

- 代码可以提前完成，功能状态按 `experimental -> beta -> stable` 逐步提升。
- 每轮只讲一个用户问题，不一次把所有能力塞进首发文案。
- 更新必须伴随示例、测试、文档、兼容性或体验上的真实变化。
- 免费层证明方法有效；课程和服务承接稳定使用与业务结果。

## 每轮固定发布包

```text
GitHub release note
30-90 秒演示视频
一篇技术/商业解释
一个 before/after
一个用户可复现命令
一个下一步产品入口
```

## 不要提前透支的话题

- 首发不要同时宣传三种模式、Obsidian、剪映、封面和高客单。
- 未经真实版本测试，不宣称完美兼容剪映工程草稿。
- 真人表演换嘴型必须公开说明生产方式。
- 没有可复现证据的能力只放路线图，不写成已经完成。
